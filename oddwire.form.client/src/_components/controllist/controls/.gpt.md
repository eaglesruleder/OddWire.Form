# Doc — Form Controls

**Scope:** Subsystem
**Domain:** oddform
**Status:** Active

---

## Summary

**Goal:** Stateless leaf controls that lay out a label+field via `ControlBase` and emit `onChange(value, key, subkey?)` (each leaf passes its own `param` as the key).

**Lifecycle:**
1. Parent passes props (`value` + `onChange`)
2. ControlBase gates `hidden`, builds the row class, renders label + field
3. Leaf renders its react-bootstrap field, spreading `{...props}` into ControlBase
4. User edits → `onChange(value, props.param)` back to the parent (param supplied as the `key`)

**Rules:**
- Controls are stateless — value comes from props
- Renders `id`/`htmlFor` = `param`; duplicate `param`s would collide in the DOM (accepted — handled later if it becomes a nuisance)
- `hidden` is handled once in ControlBase (ControlText and ControlCheckbox gate themselves — they don't use ControlBase)
- `readonly` is handled once in ControlBase (so every leaf that wraps it inherits it): `true` → the value as static text, a function → its own node in place of the field
- ControlTextField stores the raw string — no parse/normalise

**State:** `Stored: none — value is owned by the parent`

---

## Implementation

### ControlBase
Label+field row skeleton. Takes the layout slice of `CoreControlProps` (`label`/`hidden`/`stacked`/`value`/`readonly`) + `className` + `labelClassName` + `labelStyle` + `children`.
- `hidden` → render nothing
- `rowClass` = `flex` + (`column` | `items-center`) + `gap` + `mb-3` + `stack-sm` + `className`
- `readonly` → field is `readonly(props)` (function) or a `.control-static` value span (`true`); else `children`
- Renders label (`.control-label` + `labelClassName`, inline `labelStyle`) then field (`.fill`)

`CoreControlProps.readonly = boolean | (props => ReactNode)` (function param fixed to the unknown-valued props so it stays invariant across `TValue` and rides `{...props}`). `ControlButton` takes optional `variant`/`size`/`className` so callers can compose small outline icon-buttons.

### ControlText
Static, not ControlBase: label → `.separator` heading, value → `.control-static` dump. Own `hidden` gate. `className` opts into a block style (e.g. `.bubble`).

### ControlTextField
Single-line input via ControlBase. `inputMode` from `keyboardType ?? keyboardForValueType[valueType]`. Stores the raw string.

### ControlTextArea
Multi-line input; forces `stacked` (spread `{...props}` then `stacked`).

### ControlCheckbox
Own flex row (not ControlBase): label fills left, box content-width right. Own `hidden` gate.

### ControlImage
Static image via ControlBase (`stacked`): `value` (a URL/URN) → `<img src>` (max-width 100%); empty → muted `placeholder ?? "No image"`. Display-only — no `onChange` (capture/upload is future). Pairs with `dbOptions.fill` writing an image-URL column into the param.

### ControlRadio
Maps `controls` to inline radios; `checked` by `value === option.value`; shared `name = param`. `disabled` or zero options → `placeholder ?? "No Options"` muted text instead of radios. Options may come from static `controls` or, resolved upstream in `ControlItem`, from `dbOptions` (lookup).

### ControlDropdown
`Form.Select` over `controls`, empty option shows `placeholder ?? "Select…"`; `disabled` greys it. Options are static `controls` or `dbOptions`-resolved (see `lookup/` in the controllist doc). `dbOptions` config `{table, valueParam, labelParam?, filter?, joinOptions?, fill?}` (or a bare table-name string) is authored on the def; the resolver saves `valueParam`, displays `labelParam`. `fill:true` → `ControlItem`'s `fillOnChange` also writes every other column of the selected row into its matching param (record fill).

## Layout controls — `layout/`

Structural controls that own local UI state and render nested `ControlList`s over the **same flat instance + same root `onChange`**. They store **nothing** in the instance — `param` is identity only; open/active is local `useState`. All three share one config shape: `{type, param, label?, controls: ControlDef[]}` where `controls` = child controls (not options). A hidden `tab` is filtered from its tabset (by `ControlList` / `FormPage`); collapsible/popup gate their own `hidden`. Sticky headers cascade by nesting `depth` (threaded through `ControlList` → `ControlItem` → layout controls): `stickyTop(depth) = 60 + depth*48` sets each collapsible header / inline tab-bar `top`, so nested sticky bars stack instead of overlapping. A collapsible/inline-tab advances child depth by 1; a root tabset resets child depth to 0 (its bar is the fixed floor); a popup resets to 0 (its own StripLayout masthead).

### ControlCollapsible
Panel-wide box. Header `<button>` toggles a local `expanded` (`useState(false)`); header is `sticky top:60px` so it parks under the masthead on scroll. Expanded → renders children via `ControlList`.

### ControlTab
A `tab` control = one section (`label` + `controls`). A **run of adjacent `tab` controls = one tabset** (grouped in `ControlList`, or by `FormPage` at root). Takes derived `sections: {param,label,controls?,content?,disabled?,notice?}[]` + `pageLayout?` + `defaultParam?`, owns local `activeParam` (defaults to `defaultParam` if it names an enabled section, else the first enabled one). `notice` → a `ControlError` banner above the section body. A section renders `content` (a rendered node — used by programmatic callers like the DB Manager) if present, else `ControlList` over `controls` (needs `instance` + `onChange`, both optional now). `disabled` sections grey out (`.tab:disabled`) and can't be activated. JSON tabs still use `controls`; `content`/`disabled` are runtime-only, so the jsonable path is unchanged.
- `pageLayout` false (default) — inline: bar `sticky` (depth offset), wraps (`flex-wrap`); dispatched by `ControlList`/`ControlItem`.
- `pageLayout` true — root: bar `fixed bottom:0` (`height-footer`), strip-width, tabs `flex:1` split N even parts; wrapper `margin-bot-footer` clears the fixed bar. Passed only by `FormPage` when `controls[0].type==='tab'`.

### ControlPopup
`ControlButton` opens a `react-bootstrap` `Modal` (free backdrop/Escape/focus-trap). Body reuses `StripLayout` as the banner — `title=label`, `left`=a `<button>` back-arrow that closes (no `leftLink` → StripLayout renders the raw node). Dialog sized off tokens: `width: min(100vw, --strip-max) - 24px`, `height: 100svh - 100px`. Generalised like `ControlTab`: JSON popups pass `controls` (→ `ControlList`); programmatic callers may pass rendered `content` + a header `right` node + `triggerVariant`/`triggerSize` (instance/onChange optional). The DB Manager "Import" popup uses this — `right`=a 📁 file-import icon, `content`=the form catalogue.

## Styling

Keyword CSS in `public/style/`:
- `alignment.css` — `.flex`, `.flex.column`, `.flex.items-center`, `.flex.gap`, `.fill`, `.center`; 12-col grid: `.grid` (`repeat(12, minmax(0,1fr))` + gap, `.grid.items-end`, kills child `mb`), `.col-1`–`.col-12` (`grid-column: span N`), responsive overrides `.sm-col-N` (≥480px) / `.lg-col-N` (≥768px). **`.control-grid`** — the wrapper `ControlList` renders every list into: a 12-col grid whose children are **full-width by default** (`> * { grid-column: 1 / -1 }`), so a control opts into columns via its JSON `cellClassName` (e.g. `"sm-col-12 lg-col-8"`). `cellClassName` + `stacked` are on `ControlDefBase` (JSON-authorable) and land on the `ControlBase` row; `className` is runtime-only.
- `text.css` — `.bold`, `.semibold`, `.accent`, `.error` (dark red), `.text-center`, `.left`, `.right`
- `controls.css` — `.control-label`, `.control-static`, `.separator`, `.bubble`, `.stack-sm` (responsive stack < 480px)
- `theme.css` / `layout.css` — palette tokens + page shell

Colocated CSS: `layout/layoutControls.css` (imported by the layout controls) — `.collapsible*`, `.tab*`/`.tabset*`/`.tab-bar-{inline,root}`, `.popup-dialog`/`.popup-content`.
